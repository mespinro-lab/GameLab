### S1-02 — Efectes passius de branch tech s'apliquen dues vegades si unlock + event coincideixen
**Reported by**: playtester-tycoon (TYC-01)
**Tipus**: Logic bug

**Descripció**: `unlockBranchTech()` a `game.js:194` no té guard d'idempotència. El flux doble és:
1. Jugador executa `performDiscoveryAction()` → `getEligibleBranchTechs()` troba `bt_agulla_os` → `unlockBranchTech(bt_agulla_os)` s'executa, afegint l'ID al Set i aplicant `passive_effect`.
2. En el mateix cicle hi havia un event pendent (`ev_desc_agulla`). El jugador tria l'opció de descoberta → `resolveDiscoveryOption()` crida `unlockBranchTech(bt_agulla_os)` de nou.
3. El `Set.add()` és idempotent (no duplica l'ID) però `addLog()` i tots els blocs `if (pe.type === ...)` s'executen una segona vegada.

Efecte concret: `bt_agulla_os` → +6 Salut (en comptes de +3); `bt_calendari_natural` → +4 Provisions (en comptes de +2); `bt_domini_terra` → +4 Salut (en comptes de +2).

**Proof**: `game.js:194-203` — cap early-return a l'inici de la funció.

**Fix**:
```js
function unlockBranchTech(bt) {
  if (state.character.unlockedBranchTechIds.has(bt.id)) return; // afegir aquesta línia
  ...
}
```

### S1-03 — Múltiples branch techs amb prerequisits d'inclinació estructuralment inalcançables en Gen 1
**Reported by**: playtester-optimizer (OPT-15), playtester-speed-runner (SR-03, SR-04, SR-05)
**Tipus**: Balance / design bug

**Descripció**: Amb INERTIA_FACTOR=2.0 i LIFE_EXPECTANCY=14, les fórmules d'acumulació d'inclinació fan que diverses branch techs siguin impossibles d'aconseguir en Gen 1 per a jugadors que segueixen els incentius normals del joc:

**a) Intel·lecte max 0.05 — tancat per joc normal (SR-03)**
`bt_coneixement_plantes`, `bt_llavor_selectiva` i `bt_calendari_natural` requereixen `intel·lecte ≤ 0.05`. `act_tallar_pedra` (la font principal de Provisions en Gen 1) dóna `intel·lecte +0.02` per ús. Simulació:
- Ús 1: intel·lecte = 0.020
- Ús 2: intel·lecte = 0.039
- Ús 3: intel·lecte = 0.058 → **gate tancat permanentment**

Qualsevol jugador que crafi 3 vegades (el que és inevitable per comprar accions de branch tech) ha perdut accés a 3 branch techs sense rebre cap avís.

**b) Branca Místic — prerequisit impossible en Gen 1 (OPT-15)**
`bt_pintura_rupestre` requereix `espiritualitat ≥ 0.30 AND sociabilitat ≥ 0.20`. L'única acció base que genera espiritualitat és `act_ritual_foc` (+0.03, amb inèrcia). Per arribar a 0.30 calen aproximadament 18-22 cicles — LIFE_EXPECTANCY és 14. La branca Místic és estructuralment de Gen 2+, però el joc no ho comunica.

**c) bt_punta_llanca impuls 0.25 (SR-04)**
La millor acció base per impuls és `act_espiar_ramat` (+0.02). Simulació de 14 cicles exclusius: màxim reachable ≈ **0.220**, per sota del 0.25 requerit. Només accessible via event aleatori (`ev_desc_llancador`), no per acció deliberada.

**d) bt_buri intel·lecte 0.25 (SR-05)**
Requereix 15-16 cicles de crafting exclusiu per arribar a 0.25. Impossible en 14 cicles sense `bt_rasclador_fi` de stepping stone.

**Fixes suggerits**:
- `bt_pintura_rupestre`: baixar threshold de `espiritualitat ≥ 0.30` a `≥ 0.20`, o afegir una acció base espiritual de +0.05.
- `bt_coneixement_plantes` i derivats: pujar el threshold de `intel·lecte ≤ 0.05` a `≤ 0.15` o `≤ 0.20`.
- Documentar explícitament al GDD quines branches requereixen múltiples generacions per activar-se.

#### S2-02 — `getBranchTechMaturity()` selecciona la branch tech equivocada quan n'hi ha múltiples elegibles
**Reported by**: playtester-speed-runner (SR-08)
**Tipus**: Logic bug — algoritme

`performDiscoveryAction()` selecciona la "millor" tech via `getBranchTechMaturity()` (game.js:184-191). L'scoring és: per condicions `min`: `val - min`; per condicions `max`: `max - val`. Biaix sistemàtic: una tech amb condició `max: 0.05` i valor actual 0.0 puntua `0.05`. Una tech amb condició `min: 0.25` i valor actual 0.26 puntua `0.01`. La tech amb condicions `max` sempre guanya independentment de l'esforç del jugador.

Exemple concret al cicle 2: `bt_coneixement_plantes` (AND: intel·lecte max 0.05 + impuls max 0.10) puntua 0.15, derrotant `bt_rasclador_fi` (OR: impuls max 0.10) que puntua 0.10 — el jugador no tria i no rep cap indicació de quina tech serà seleccionada.

**Fix**: (a) Mostrar al jugador quina tech serà desblocat abans de confirmar, o (b) permetre al jugador triar entre totes les elegibles.

#### S2-05 — `randInt(min, max)` comportament indefinit si `outputMinBonus > output_max - output_min`
**Reported by**: playtester-tycoon (TYC-02)
**Tipus**: Logic bug (latent)

`randInt` a `game.js:512`:
```js
return Math.floor(Math.random() * (max - min + 1)) + min;
```
Si `output_min + outputMinBonus > output_max`, `max - min + 1` és negatiu i la funció pot retornar valors fora del rang esperat. No és reachable amb el `data.js` actual (bt_rasclador_fi dóna bonus 1 a acció amb output_min:2, output_max:4 → efectiu min:3, max:4 — valid), però no hi ha cap guard defensiu.

**Fix**: Afegir al principi de `randInt`: `if (min > max) return min;`

#### S2-07 — Modal de successió sense null-guard a `successor.inheritedInclination`
**Reported by**: playtester-tycoon (TYC-04)
**Tipus**: Latent crash risk

`renderModals()` a `game.js:1096`:
```js
const dominantAxis = AXES.reduce((a, b) =>
  Math.abs(successor.inheritedInclination[a]) > Math.abs(successor.inheritedInclination[b]) ? a : b
);
```
Si `successor.inheritedInclination` és `undefined`, la línia llança TypeError i el modal de successió queda en blanc. El flux actual és segur (tots els fills passen per `triggerSuccession()` i sempre obtenen `inheritedInclination`), però qualsevol futur codi que afegeixi objectes bare `{id, label}` al `siblingPool` causaria un crash silenciós en el moment més crític del joc.

**Fix**: Afegir guard: `const incl = successor.inheritedInclination ?? {};` i usar `incl` en comptes de `successor.inheritedInclination`.

#### S2-08 — `act_tenir_fills` no té early-return a `executeAction()` — explotable per consola
**Reported by**: playtester-tycoon (TYC-05)
